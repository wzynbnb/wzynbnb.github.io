类加载器（前置知识）

概念：

当程序要使用某个类时，如果该类还没有被加载到内存中，则系统会通过类的加载，类的连接，类的初始化这三个步骤来对类进行初始化，如果不出现意外情况，JVM会连续完成这三个步骤，所以有时也把这三个步骤统称为类加载或者类初始化

三个步骤

类加载

将class文件读入内存，并为之创建一个java.lang.Class对象，任何类被使用时，系统都会为之创建一个java.lang.Class对象

类的连接

验证阶段：用于检验被加载的类是否有正确的内部结构，并和其他类协调一致

准备阶段：负责为类的类变量分配内存，并设置默认初始化值

解析阶段：将类的二进制数据中的符号引用替换为直接引用

类的初始化

主要就是为类变量进行初始化

步骤：

  • 加入类还未被加载和连接，则程序先加载并连接该类
  • 加入该类的直接父类还未被初始化，则先初始化其直接父类（也是按照这几个步骤）
  • 加入类中有初始化语句，则系统一次执行这些初始化语句

类初始化的时机：

  • 创建类的实例时
  • 调用类的类方法时
  • 访问类或接口的类变量，或者为该类变量赋值时
  • 使用反射方式来强制创建某个类或接口对应的java.lang.Class对象时
  • 初始化某个类的子类时
  • 直接使用java.exe命令来运行某个主类时

范式

第一范式

1NF

每一个列都是不可分割的原子项

不满足第一范式，因为系可以分成两个小列

满足了第一范式

第二范式

在第一范式的基础上，消除非主属性对主码的部分函数依赖

比如在下面表的结构中，学号+课程名称可以得到分数，学号可以得到姓名，姓名可以得到系名，系名可以得到系主任，由于传递依赖，学号可以得到姓名、系名、系主任，所以，这个表中，只要知道了学号和课程名称，就可以知道任何列的值，故学号+课程名称为码，学号、课程名称为主属性，其他属性为非主属性。

码可以得到分数，使用了学号+课程名称，也就是码中的全部属性，这叫完全函数依赖

码可以得到姓名、系名、系主任，但只使用了学号，也就是没有用到码中的全部属性，这叫部分函数依赖

第二范式就是要消除部分函数依赖，让表中，码得出任何属性，都是属于完全函数依赖，即用到了码中的所有属性

满足了第二范式

第二范式可以解决数据冗余的问题

第三范式

在第二范式的基础上，任何非主属性不依赖于其他非主属性（在2NF的基础上消除传递依赖）

数据库连接池

传统的 JDBC 访问数据库，每次都需要连接数据库，访问完成之后再断开连接。每次都是这样，这就降低了效率，增加了开销。

数据库连接池是在系统初始化的时候，创建一个容器，称为池子，存放着很多的数据库连接对象，当程序需要对数据库进行操作时，不再单独创建数据库连接对象，而是直接从容器中取，使用完之后也不再释放资源，而是归还容器池。

这样就提高了效率，增加了数据库连接对象的利用率

JSP

概述

JSP 全称 Java Server Pages （ java 服务器端页面）

JSP 是一种特殊的页面，其中既可以写 HTML 标签，又可以写 Java 代码，可以达到简化书写的目的

如果通过 Servlet 获取一个页面，其中大部分是静态 HTML ，只有非常小的部分是动态资源，那么 Servlet 需要生成一个包含非常多 HTML 代码的响应体

使用 JSP ，就可以很轻松的实现同样的效果

原理

为什么 JSP 也可以被浏览器访问到，并获取动态资源呢，因为 JSP 本质就是一个 Servlet

Tomcat 会把运行的 JSP 文件生成的 .java .class 文件放在 /work 目录下

可以看，jsp 生成的 ,java 文件中，这个类是继承了另一个类的，而这个类，继承了 HttpServlet，所以本质 JSP 就是一个 Servlet，它也有自己的 Service 方法

JSP 把本应你再 Servlet 中写到响应体中的 HTML 代码，帮你封装到它的 Service 方法中了，这就达到了简化书写的效果，非常赞

MVC 开发模式

为什么要学 MVC 开发模式？

JSP 演变历史：

  1. 一开始只有 Servlet，返回给用户的页面只能通过 response 对象传输，也就是 HTML 标签都需要写在 Servlet 中，非常麻烦
  2. 后来有了 JSP ，简化了 Servlet 的开发
  3. 在后来人们过度使用 JSP 开发动态页面，导致页面里又有 HTML 标签，又有 JS 代码，又有 Java 代码，非常的难以阅读，难以维护，难以分工合作
  4. 最后 Java 的 web 开发借鉴 MVC 开发模式，思想是：让该在哪里的代码就在哪里。使得程序的设计更加合理

MVC：

  1. M：Model，模型。使用 JavaBean 来实现
    1. 完成具体业务操作，比如查询数据库，封装对象等等
  2. V：View，视图。使用 JSP 来实现
    1. 展示数据
    2. 如果在 MVC 开发模式中，需要在 JSP 里写 Java 代码，要用下面的技术来代替
      1. EL 表达式
      2. JSTL 标签
  3. C：Controller，控制器。使用 Servlet 来实现
    1. 获取用户的输入
    2. 调用模型
    3. 将数据交给视图进行展示

MVC 优缺点：

  1. 优点
    1. 耦合性低，方便维护和分工开发
    2. 重用性高
  2. 缺点
    1. 使得项目架构变复杂，对开发人员的要求提高了

会话技术

概述

会话：一次会话包含多次请求和响应，从浏览器第一次给服务器发送请求时建立，直到有一方断开为止

会话的功能：在一次会话的范围内（多次请求间），共享数据

方式：

  1. 客户端会话技术：Cookie
  2. 服务器端会话技术：Session

Cookie

服务器创建 Cookie 对象，发送给浏览器，浏览器收到 Cookie 后，以后发送请求，就会携带 Cookie 对象，使服务器又可以接收到

使用步骤：

  1. 创建 Cookie 对象，绑定数据

原理

在第一次请求、响应时，服务器端发送的响应头中会加一个键值对

set-cookie:msg=hello

浏览器收到这个响应，根据 HTTP 协议的约束，他会将这个数据保存起来，等下次发送请求的时候，放到自己的请求头中

cookie:msg=hello

服务器端收到了带着这个数据的请求头，也就可以通过 Tomcat 封装好的方法来获取 Cookie 对象了

特点

  1. Cookie 存储数据在客户端浏览器，这就导致了数据的安全性问题和稳定性问题
  2. 浏览器对单个 Cookie 的大小有限制，约 4KB ，以及对同一个域名下的总 Cookie 数量也有限制，约 20 个

作用

  1. Cookie 一般用于存储少量的不太敏感的数据
  2. 最重要的一个作用是：在不登陆的情况下，完成服务器对客户端的身份识别
    1. 举例：不登陆百度账号的情况下，对百度的搜索进行个性化设置，然后关闭浏览器，再打开，依然会保存着上一次做的设置
    2. 原因：保存了个性化设置的时候，浏览器向服务器发送一个请求，服务器返回一个带 Cookie 的响应，浏览器收到后保存 Cookie，下一次打开浏览器，浏览器向服务器发送带 Cookie 的请求，服务器就知道你做过这些设置，就给你应用上了

Cookie 的生命周期

默认情况下，浏览器关闭，Cookie 也就死亡了

    • 负数：就是默认情况，关闭浏览器就删除 Cookie
    • 零：直接让浏览器删除这条 Cookie 信息

Session

概述

Session 是服务器端会话技术，在一次会话的多次请求间共享数据，将数据保存在服务器端的对象中

Session 是依靠 Cookie 实现的，并且在多次请求中，获取到的 Session 都是同一个

Session 其实就是将响应头写 Cookie 的位置，写了一个参数 JSESSIONID=xxxxxxxxx

细节

1.当客户端关闭后，服务器不关闭，两次获取的 Session 默认情况下不是同一个，因为客户端关闭后，会话断开，服务器端想要在获取 Session 是获取不到的，再打开浏览器，服务器端获取 Session 还是没有，然后会创建一个新的 Session ，这就导致再获取到的 Session 和第一次获取的不是同一个了

如需浏览器关闭、开启后获取的 Session 都是同一个，只需要修改 Cookie 的存活时间就好了

**2.**客户端不关闭，服务器关闭后，两次获取的 Session 是同一个吗？

不是同一个，服务器关闭再开启，内存中的数据都丢失了。但是可以通过 Sessioin 的钝化和活化来确保 Session 中的数据不丢失（但 Session 的 ID 肯定会有变化）

Session 钝化：在服务器正常关闭之前，将 Session 对象序列化到硬盘中

Session 活化：在服务器启动后，将 Session 文件转化为内存中的 Session 对象

钝化和活化是相反的操作

本地的 Tomcat 服务器是会自动进行 Session 的钝化和活化的，所以使用本地的 Tomcat 服务器（也就是通过 startup.bat 启动），并把 web 项目部署到本地的 Tomcat 服务器的话，关闭服务器前后，获取到的 Session 中数据是相同的

但是 IDEA 中不能实现，因为虽然 IDEA 的 Tomcat 也进行钝化和活化的过程，但是在活化前，IDEA 会删除存放 Session 文件的文件夹，然后再创建一个新的，这就使得活化失败，所以 IDEA 不能完成这种操作

**3. **Session 何时被销毁

  1. 服务器关闭时

  2. Session 对象调用 invalidate()

  3. Session 默认失效时间为 30 分钟，这个值可以在 Tomcat 的 /conf/web.xml 里修改

    1. <session-config>
    <session-timeout>30</session-timeout>
</session-config>
